# MCP Tools Implementation Plan

> Superpowers skills were not available in this Codex session, so this records the same writing-plans/TDD/verification loop for the `mcp-tools` module.

**Goal:** Build the `mcp-tools` module so the Search stage can call external `search` and `fetch` tools through an MCP client wrapper with explicit timeout control and structured pydantic outputs.

**Architecture:** Create a `deep_research.mcp_tools` package. `models.py` owns `SearchHit`, `FetchedDoc`, and normalized tool-call models. `transport.py` owns the MCP SDK adapter around `ClientSession.call_tool`. `tools.py` owns the public `MCPTools.search` and `MCPTools.fetch` API plus retry/timeout behavior. Tests use a fake MCP transport to verify behavior without network or real servers.

**Tech Stack:** Python 3.14, asyncio, MCP Python SDK, pydantic, pytest, pyright strict, ruff.

## File Structure

- Create `src/deep_research/mcp_tools/models.py`: pydantic result and call models.
- Create `src/deep_research/mcp_tools/transport.py`: `MCPToolTransport` protocol and `MCPClientSessionTransport`.
- Create `src/deep_research/mcp_tools/tools.py`: public search/fetch wrapper with explicit timeout and finite retry.
- Create `src/deep_research/mcp_tools/__init__.py`: public exports.
- Create `tests/mcp_tools/test_tools.py`: acceptance tests for §5 invariants and §8 behavior.
- Modify `docs/guide/01-build-order.md`: mark `mcp-tools` cells complete after T1 gate passes.

## Acceptance Mapping

- Search returns structured hits: `tests/mcp_tools/test_tools.py::test_search_returns_structured_hits`.
- Fetch returns document content: `test_fetch_returns_document_content`.
- Text-only fetch normalization: `test_fetch_can_normalize_text_only_tool_result`.
