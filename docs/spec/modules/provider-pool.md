# provider-pool 规约 —— 多 Provider 号池

## 1 职责　（强）
对多个 LLM 提供商、每个提供商下的多个 API Key 做池化管理与调度，对外提供"发起一次 LLM 调用"的能力。通过调度与限流规避上游 RPM 与并发限制，支撑多路子 Agent 并发扇出不中断。

## 2 边界与非目标　（强）
- 不做 prompt 组装、不做输出 schema 校验 —— 交给 [[subagent]]。
- 不做编排 —— 交给 [[kernel]]。
- 不决定预算上限策略，仅上报每次调用的 token/成本（预算池语义见 03-cross-cutting.md）。

## 3 关键接口/契约　（中）
- `acquire() -> 选中的 Key`：按**最少并发优先 + 加权轮询**挑一个当前可用、未熔断、未触发滑窗限流的 Key。
- `call(request) -> response`：用选中的 Key 发起调用，返回结果并附 token/成本元数据；内部处理重试/熔断。
- 按 **Key 维度**做滑动窗口限流。

## 4 数据模型　（中）
- `ProviderConfig`：提供商及其 Key 列表、各 Key 的 RPM/并发上限、权重。
- `KeyState`：单 Key 的实时状态（当前并发数、滑窗内计数、熔断状态与恢复时间）。
- `LLMRequest` / `LLMResponse`：请求载荷与含 token/成本/状态码的响应。

## 5 不变式　（强）
1. 调度优先选当前并发数最少的可用 Key（同级再按权重轮询）。
2. 处于熔断状态的 Key 不会被 `acquire` 选中。
3. 滑动窗口限流保证单 Key 在窗口内调用数不超过配置 RPM。

## 6 依赖　（强）
- 模块：cross-cutting（config、token 计费类型）。
- 库：`httpx`。

## 7 失败语义（弹性接缝）　（中）
- HTTP 429 或超时 → 该 Key 熔断、请求**换道**到其他可用 Key，并按**指数退避**重试。
- 所有 Key 均不可用 → 上抛**致命错误**（不静默降级）。
- 这是允许做错误处理的接缝之一；调用方不再为此包防御性 try。

## 8 验收标准　（强）
- **given** 某 Key 返回 429，**when** 再次调度，**then** 该 Key 在熔断窗口内不被选中、请求换到其他 Key 继续。
- **given** 多个并发调用，**when** 调度，**then** 各 Key 的当前并发计数随获取/释放正确增减。
- **given** 单 Key 短时间内调用逼近 RPM，**when** 继续请求，**then** 滑窗限流阻塞/排队该 Key 的新请求。
- **given** 多个等权/不等权 Key，**when** 大量调度，**then** 分布符合最少并发优先 + 权重。

## 9 开放决策　（——）
- 指数退避的基数/上限/抖动策略。
- 熔断阈值（连续失败数）与半开恢复策略。
- 加权轮询的具体算法。
- 是否每 provider 维护独立 `httpx` client 与连接池。
