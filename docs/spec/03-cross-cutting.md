# 横切关注

地基层能力，被所有模块共享。本文件给约定与数据模型意图，精确签名交由实现者（见各模块 §9）。

## 配置

- **来源**：Provider/Key 等敏感配置从环境变量与本地配置文件加载，禁止硬编码进代码。
- **模型**：用一个 pydantic `Settings` 模型聚合配置（providers 列表、每 provider 的 keys、各 Key 的 RPM/并发上限、token 预算上限、默认 model、超时阈值）。字段为意图描述，精确定义见 [[provider-pool]] §4 与实现者决策。
- **校验**：配置在启动时一次性加载并经 pydantic 校验；缺失必填项属**致命错误**，fail-loud 直接终止，不静默用默认值兜底。

## 可观测

结构化日志（建议 JSON 行），统一在以下打点位置产生事件：

- 每次 agent 调用：开始/结束、命中 Journal 与否、token 消耗、耗时。
- 每次 provider 调用：选中的 Key、是否重试/熔断、状态码、耗时。
- 每个阶段边界（Plan/Search/Verify/Write）：进入/退出、条目数。

日志须可按"一次研究任务"的关联 id 串联，便于事后复盘时延与成本。

## 错误分类 (error taxonomy)

与编码约定的"异常处理只在弹性接缝"纪律配套，错误分两类：

- **可重试 (retryable)**：HTTP 429、超时、子 Agent 输出 schema 校验失败。处理点见各弹性接缝（[[provider-pool]] / [[subagent]]）。
- **致命 (fatal)**：配置缺失、全部 Key 不可用、预算池耗尽且无法继续。这类错误**fail-loud 上抛**，由顶层 runner 记录后终止——因为 [[journal]] 让恢复成本近乎为零，响亮崩溃优于静默降级。

类型上建议用一个错误基类区分 `Retryable` 与 `Fatal` 两支，便于弹性接缝做模式匹配。

## token 计费

- **计费类型**：每次 provider 调用产出"输入/输出 token 数 + 估算成本"，挂在 [[provider-pool]] 的响应上。
- **共享预算池**：全局累计 token/成本，由一个预算池对象持有；[[subagent]] 在调用前后结算。
- **超限熔断**：累计消耗达到 `Settings` 配置的预算上限时，预算池拒绝后续扣减请求，触发 fatal 终止当前任务。这是成本治理的硬闸门（预算上限可配置）。
